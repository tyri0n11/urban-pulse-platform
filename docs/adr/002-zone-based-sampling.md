# ADR 002: Zone-Based Route Sampling

**Status:** Accepted  
**Date:** 2024-Q4  
**Context:** Route network design for Urban Pulse

---

## Context

The VietMap Traffic API provides congestion data for individual road segments across HCMC. The full road network contains thousands of segments — not all are relevant for a city-wide anomaly detection system, and querying all of them every 5 minutes would be prohibitively slow and would hit API rate limits.

Two design approaches were considered:

**Option A**: Monitor individual road segments (e.g., a 200m stretch of Xa lộ Hà Nội). Maximum granularity, but requires segment-to-cluster mapping for any aggregate analysis.

**Option B**: Define geographic zones and monitor inter-zone routes. Each route aggregates the congestion signal along a logical travel corridor between two zones.

---

## Decision

Adopt **Option B** — a zone-based routing model with 6 zones and 20 inter-zone routes.

### Zone Definitions

| Zone | ID | Characteristics |
|------|-----|----------------|
| Urban Core | zone1 | District 1, 3 — CBD, highest density |
| Eastern Innovation | zone2 | District 9, Thu Duc — tech parks, university |
| Northern Industrial | zone3 | District 12, Binh Chanh — factories, logistics |
| Southern Port | zone4 | District 4, 7 — ports, export processing zones |
| Western Periurban | zone5 | Binh Tan, Hoc Mon — residential, commuter origin |
| Southern Coastal | zone6 | Can Gio, Nha Be — low-density, coastal |

### Route Format

```
zone{N}_{zone_name}_to_zone{M}_{zone_name}
```

Example: `zone1_urban_core_to_zone4_southern_port`

Routes are asymmetric (origin → destination), directional. The 20 routes are defined in `routes.json` (source of truth, generated by `routes_generator.py` from `zones.json`).

---

## Consequences

### Benefits

**Operational tractability**: 20 routes is manageable for a single analyst to interpret. A heatmap of 20 routes is readable; a heatmap of 3000 road segments is not.

**Alignment with travel demand**: Commuters travel between zones (home → work → home), not between arbitrary segments. Zone-to-zone routes match the mental model of city planners and traffic operators.

**Stable identifiers**: Zone IDs are stable across API versions. Segment IDs may change if the map provider updates their road network. Route IDs like `zone1_urban_core_to_zone4_southern_port` are human-readable and self-documenting.

**IsolationForest scope**: Training one IsolationForest per route (20 models) is feasible on a single machine. One model per segment would require 3000+ models, each with fewer samples per route.

**Aggregation for ML**: Gold-layer features (`avg_duration_minutes`, `avg_heavy_ratio`) are computed at the route level. Zone-based routes provide a natural aggregation boundary that reduces within-route noise while preserving inter-route variation.

### Costs

**Loss of spatial resolution**: A route-level anomaly cannot pinpoint which sub-segment is the bottleneck. The `/explain` endpoint partially compensates by reporting `max_severe_segments` (the count of severely congested segments on the route) as supporting evidence.

**Route definition subjectivity**: Zone boundaries were drawn heuristically based on administrative districts and functional land use. A different analyst might draw different boundaries. The zones are documented in `zones.json` with lat/lon bounding boxes for reproducibility.

**20 routes may miss corridors**: HCMC has radial arterials (Xa lộ Hà Nội, QL13, etc.) that cross multiple zones. A route from zone1 to zone2 implicitly covers Xa lộ Hà Nội but does not distinguish it from parallel routes. Future work could add named-corridor routes.

---

## Alternatives Rejected

**Individual segment monitoring**: Too many identifiers, too many models, no stable ID contract with the API provider.

**Administrative district monitoring**: HCMC has 22 districts — monitoring district pairs would produce 22×22 = 484 combinations, most of which have no meaningful direct travel demand. Zone-based sampling focuses on the 20 corridors with real commuter volume.

**Single city-wide model**: A single IsolationForest trained on all routes conflates route-specific normal behaviour. Zone1→Zone4 (port corridor) has different congestion patterns than Zone2→Zone3 (industrial commuter). Per-route models are strongly preferred.
